# Research: Axiom Critical Network Resilience

## Scope
This note captures only the evidence needed to design the first implementation pass for `axiom-critical-network-resilience`.

## Current Runtime Evidence
- `autossh-reverse-ssh.service` is a system service with `Restart=always`, but currently has `OOMScoreAdjust=0`, `MemoryMin=0`, and `MemoryLow=0`.
- `cloudflared.service` is a system service with `Restart=on-failure`, but currently has `OOMScoreAdjust=0`, `MemoryMin=0`, and `MemoryLow=0`.
- `clash-verge.service` is a system service generated by `programs.clash-verge.serviceMode`; it runs `clash-verge-service` and child `verge-mihomo`, but currently has `OOMScoreAdjust=0`, `MemoryMin=0`, and `MemoryLow=0`.
- `app-clash\x2dverge@autostart.service` is generated by `systemd-xdg-autostart-generator` from `/run/current-system/sw/etc/xdg/autostart/clash-verge.desktop`; it runs the GUI/WebKit wrapper and currently reports `OOMScoreAdjust=200`.
- `user@1000.service` starts the user manager with upstream `OOMScoreAdjust=100`, which contributes to user-session processes being easier to kill than system services.
- `cloudflared` currently exposes a local readiness endpoint at `http://127.0.0.1:20241/ready`, but the dotfiles do not explicitly declare `metrics = 127.0.0.1:20241` in the cloudflared config.
- The machine currently has about 46 GiB RAM and about 8.8 GiB disk swap.

## Source Evidence
- `hosts/axiom/default.nix` defines `autossh-reverse-ssh.service` and the `modules.services.cloudflared` host config.
- `modules/services/cloudflared.nix` defines the reusable cloudflared system service.
- `modules/desktop/apps/clash-verge.nix` enables `programs.clash-verge` with `serviceMode`, `tunMode`, and `autoStart`.
- Nixpkgs `nixos/modules/programs/clash-verge.nix` generates `systemd.services.clash-verge` when `serviceMode` is enabled.
- NixOS systemd units support `overrideStrategy = "asDropin"`; this can create a drop-in for a generated or externally provided unit.
- NixOS `programs.ssh.knownHosts` writes pinned host keys to `/etc/ssh/ssh_known_hosts`, which avoids mutating the user's `~/.ssh/known_hosts`.
- NixOS `zramSwap` can enable compressed in-memory swap with a configurable priority higher than disk swap.

## Design Implications
- Kernel OOM selection must be addressed with `OOMScoreAdjust`; cgroup memory protection alone is not enough for the failure mode observed.
- `MemoryMin` and `MemoryLow` are still useful as reinforcement for cgroup-aware reclaim and systemd-oomd behavior.
- Strong protection should apply to system services, especially `sshd`, `autossh`, `cloudflared`, and Clash service-mode/core.
- Clash GUI should not be more protected than the core, but it should stop being unnecessarily penalized.
- A long-term autossh remote health check should pin the remote host key declaratively instead of scanning and trusting on every run.
- The autossh check should prove the reverse endpoint reaches `axiom`, not only that remote port `2223` is listening. Comparing the key returned by remote `ssh-keyscan -p 2223 127.0.0.1` against `axiom`'s local SSH host public key is stronger than accepting a generic SSH banner.

## Remaining Implementation Checks
- Verify the exact Nix attribute name and escaping for the user drop-in targeting `app-clash\x2dverge@autostart.service`.
- Verify whether a user unit can apply `OOMScoreAdjust=0` after setting `user@1000.service` to `OOMScoreAdjust=0`; if not, keep the GUI protection to `MemoryLow` plus restart behavior and document the limitation.
- Verify whether `cloudflared` accepts an explicit `metrics = "127.0.0.1:20241"` in the generated JSON/YAML config without changing tunnel behavior.
- Verify healthcheck counter behavior with non-destructive fake inputs before relying on timers.
